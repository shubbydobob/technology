로드밸런싱이란 무엇인가?

여러 서버에 들어오는 요청을 적절히 분산해 성능, 안정성, 확장성을 높이는 트래픽 분산 방식입니다.

네트워크

로드밸런싱이란 무엇인가?

로드밸런싱은 클라이언트 요청을 하나의 서버에 몰아주지 않고 여러 서버로 나누어 보내는 기술입니다. 서버 한 대가 감당해야 하는 부하를 줄이고, 특정 서버에 장애가 생겨도 다른 서버로 요청을 보내 서비스 중단을 줄일 수 있습니다.

간단한 예시

사용자 3,000명이 동시에 쇼핑몰에 접속한다고 가정합니다. 서버가 한 대뿐이면 CPU, 메모리, DB 연결 수가 한계에 도달할 수 있습니다. 로드밸런서를 앞에 두고 웹 서버 3대를 연결하면 요청을 여러 서버로 나눠 처리할 수 있습니다.

사용자 요청
  -> Load Balancer
    -> Web Server 1
    -> Web Server 2
    -> Web Server 3

주요 구성 요소

구성설명
Client웹 브라우저, 앱처럼 서비스를 호출하는 사용자 측입니다.
Load Balancer요청을 받아 여러 서버 중 하나로 전달하는 중간 장비 또는 서비스입니다.
Backend Server실제 요청을 처리하는 WAS, API 서버, 웹 서버입니다.
Health Check서버가 정상인지 주기적으로 확인하고 비정상 서버를 분산 대상에서 제외합니다.

분산 방식

  • Round Robin요청을 서버에 순서대로 나눠 보냅니다. 서버 성능이 비슷하고 요청 처리 시간이 크게 다르지 않을 때 단순하게 쓰기 좋습니다.
  • Least Connections현재 연결 수가 가장 적은 서버로 요청을 보냅니다. 요청 처리 시간이 들쭉날쭉한 서비스에서 유리합니다.
  • IP Hash클라이언트 IP를 기준으로 특정 서버에 요청을 보내는 방식입니다. 같은 사용자를 같은 서버로 보내야 할 때 사용됩니다.
  • Weighted서버 성능에 따라 가중치를 다르게 줍니다. 성능이 좋은 서버가 더 많은 요청을 받도록 조정할 수 있습니다.

세션과의 관계

로그인 세션을 특정 WAS 메모리에만 저장하면 다음 요청이 다른 WAS로 분산될 때 로그인 정보가 없을 수 있습니다. 이를 해결하려면 세션 클러스터링, Sticky Session, Redis 같은 외부 세션 저장소를 사용합니다.

기술 포인트

  • 로드밸런싱의 핵심 목적은?트래픽을 여러 서버에 분산해 성능과 가용성을 높이는 것입니다. 단순히 속도를 높이는 것뿐 아니라 장애 서버를 제외해 서비스 중단 범위를 줄이는 역할도 합니다.
  • Health Check가 중요한 이유는?비정상 서버로 요청을 계속 보내면 사용자는 오류를 보게 됩니다. 로드밸런서는 상태 확인 결과를 기준으로 정상 서버에만 트래픽을 전달해야 합니다.
  • Sticky Session은 언제 필요한가?세션이 서버 메모리에 저장되어 있고 같은 사용자의 요청을 같은 서버로 보내야 할 때 필요합니다. 다만 서버 장애나 확장 상황에서는 외부 세션 저장소가 더 안정적일 수 있습니다.
