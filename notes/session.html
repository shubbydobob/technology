세션이란 무엇인가?

HTTP의 stateless 특성을 보완하기 위해 서버가 사용자 상태를 일정 시간 동안 보관하는 방식입니다.

백엔드

세션이란 무엇인가?

세션은 서버가 사용자의 로그인 상태나 장바구니 같은 상태 정보를 유지하기 위해 사용하는 저장 공간 또는 상태 관리 방식입니다. HTTP 요청은 기본적으로 서로 독립적이기 때문에, 서버는 세션 ID를 기준으로 같은 사용자의 요청인지 식별합니다.

간단한 예시

사용자가 로그인하면 서버는 세션 저장소에 사용자 정보를 저장하고, 브라우저에는 세션 ID가 담긴 쿠키를 내려줍니다. 이후 브라우저가 요청할 때 쿠키를 함께 보내면 서버는 세션 ID로 로그인 사용자를 찾습니다.

1. POST /login
2. 서버: 세션 저장소에 userId 저장
3. 서버: Set-Cookie: JSESSIONID=abc123
4. 다음 요청: Cookie: JSESSIONID=abc123
5. 서버: abc123 세션을 조회해 사용자 식별

세션 동작 흐름

단계설명
로그인 요청사용자가 아이디와 비밀번호를 서버로 보냅니다.
인증 성공서버가 사용자 정보를 확인하고 세션을 생성합니다.
세션 ID 발급서버는 세션 ID를 쿠키로 브라우저에 전달합니다.
인증 유지브라우저는 요청마다 세션 ID 쿠키를 보내고 서버는 세션 저장소를 조회합니다.
로그아웃/만료서버가 세션을 삭제하거나 유효 시간이 지나면 인증 상태가 종료됩니다.

구분쿠키세션
저장 위치브라우저서버 또는 외부 저장소
주요 역할세션 ID, 사용자 설정 같은 작은 값 저장로그인 사용자 정보 같은 서버 측 상태 관리

확장 이슈브라우저가 직접 보냅니다.서버가 여러 대면 공유 저장소나 세션 고정 전략이 필요합니다.

저장 위치

  • 서버 메모리구현은 단순하지만 서버가 재시작되면 세션이 사라지고, 서버가 여러 대일 때 공유가 어렵습니다.
  • DB영속성은 좋지만 요청마다 조회하면 부하가 커질 수 있습니다.
  • Redis빠른 읽기/쓰기와 TTL 관리가 가능해 분산 서버 환경의 세션 저장소로 자주 사용됩니다.

로드밸런싱과 세션

WAS가 여러 대이고 세션을 각 서버 메모리에 저장하면, 사용자의 다음 요청이 다른 서버로 가면서 세션을 찾지 못할 수 있습니다. Sticky Session으로 같은 서버에 고정하거나, Redis 같은 공통 세션 저장소를 두어 모든 서버가 같은 세션을 조회하게 만들 수 있습니다.

기술 포인트

  • HTTP가 stateless라는 말의 의미는?각 요청이 이전 요청 상태를 기억하지 않는다는 뜻입니다. 그래서 로그인 상태를 유지하려면 쿠키, 세션, 토큰 같은 별도 상태 관리 방식이 필요합니다.
  • 세션 ID는 왜 쿠키에 담는가?브라우저가 요청마다 자동으로 쿠키를 보내기 때문에 서버가 같은 사용자의 세션을 찾을 수 있습니다. 실제 사용자 정보는 서버에 두고 쿠키에는 식별자만 두는 방식입니다.
  • 분산 서버에서 세션 문제는 어떻게 해결하는가?Sticky Session으로 같은 서버에 요청을 고정하거나 Redis 같은 외부 저장소에 세션을 저장합니다. 실무에서는 장애 대응과 확장성을 고려해 외부 세션 저장소를 많이 사용합니다.
